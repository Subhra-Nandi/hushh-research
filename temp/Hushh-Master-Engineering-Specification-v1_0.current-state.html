Updated Current-State Edition for Salesforce Review | March 2026 | Hushh Research | CONFIDENTIAL

This edition keeps the founder's engineering structure and intended platform language intact while aligning the document to the current Hushh runtime and the platform direction now visible in the repository.

1. System Overview

1.1 What We Are Building

Hushh is being built as a consent-first personal-agent platform in which the individual owns the private data layer, governs access through explicit scoped approvals, and carries that authority across product surfaces, counterparties, and agents.

In the current release train, that architecture is already visible in concrete form. Kai is the flagship product surface, the world model is the private data backbone, VAULT_OWNER and scoped consent tokens form the control plane, and the Investor plus RIA rollout provides the first multi-actor workflow expression of the platform. The broader Hushh ambition extends beyond finance into a cross-domain personal-agent system, but the core principles are already operating as runtime facts rather than aspirational language.

From an engineering standpoint, Hushh should be understood as three cooperating planes: an encrypted private-data plane, a relational workflow and public-discovery plane, and an agent and service plane above both. That separation is the design center for everything else in this specification.

1.2 Platform Stack (Immutable)

2. Salesforce CRM Data Model

2.1 Core Objects

The engineering model described in Salesforce terms should be read as a mapping specification rather than a claim that private Hushh data already lives inside Salesforce CRM records. Hushh currently maintains a split architecture in which encrypted private content remains in the world model while CRM-shaped workflow, verification, and discovery state remains in relational tables.

Object Current Hushh Representation Intended Mapping
User / Contact actor_profiles backed by Firebase identity bootstrap. Canonical actor record for persona state, opt-in state, and workflow ownership.
Advisor ria_profiles, ria_firms, and ria_firm_memberships. RIA identity and firm context form the advisor-side relationship graph.
Consent consent_audit, consent_exports, and consent_scope_templates. Approval history, scope envelope, export evidence, and revocation status already exist as first-class control-plane objects.
Relationship advisor_investor_relationships and ria_client_invites. Advisor access is a consent-backed workflow relationship, not a generic account link.
Private profile / preference state world_model_data plus summarized metadata in world_model_index_v2. Private user-owned facts remain in the encrypted personal data plane and should not be flattened into CRM storage.
Domain metadata domain_registry. Provides the domain vocabulary used by discovery, scoping, and future enterprise mappings.

The same mapping leaves room for a future value ledger on top of consent. In the Hushh model, a handshake does not just authorize access; over time it can also carry economic metadata so that whenever Kai or another approved agent retrieves a governed slice of the user's world model, the event can be priced, audited, and settled without relocating the user's private record into CRM storage.

2.2 Object Relationships

Hushh's relationship model is best understood as a user-owned private core surrounded by actor, firm, consent, and marketplace relationships. The engineering benefit of this separation is that private encrypted content can evolve independently from workflow and public-discovery contracts.

actor_profiles (1) ---- (0..1) ria_profiles
actor_profiles (1) ---- (many) consent_audit
actor_profiles (1) ---- (many) world_model_data / world_model_index_v2
ria_profiles   (many) -- (many) ria_firms via ria_firm_memberships
ria_profiles   (many) -- (many) investors via advisor_investor_relationships
actor_profiles (1) ---- (0..1) marketplace_public_profiles
ria_profiles   (1) ---- (0..1) marketplace_public_profiles

That structure is what allows Salesforce-facing workflow concepts to be modeled cleanly without eroding the encrypted private-data boundary that defines the Hushh platform.

2.3 Preference JSON Schemas

Preference state in Hushh is domain-scoped rather than monolithic. The active product path currently stores onboarding continuity and persona-relevant state under encrypted financial.profile, with safe summaries and counters maintained in the world-model index. Additional actor and marketplace preferences that belong to trust or discovery surfaces remain relational.

Schema Family Current Form Engineering Rule
Private domain payloads Encrypted world-model blob Private content is written only to world_model_data.
Sanitized summaries Per-domain metadata in world_model_index_v2.domain_summaries No sensitive financial details or key material may be written into the index.
Domain discovery available_domains, computed_tags, and registry metadata Dynamic scope generation relies on this discovery layer.
Trust and workflow settings Actor, RIA, and marketplace tables These settings do not belong in the encrypted private blob unless they are truly private user-owned facts.

3. Agent Architecture (ADK)

3.1 Agent Hierarchy

Kai is implemented as a hub-and-spoke ADK-oriented runtime with explicit layering between agents, tools, operons, and services. The hierarchy matters because it keeps consent validation and private-data access from being casually bypassed by convenience code.

Layer Responsibility Engineering Constraint
Agent Coordinates behavior, role selection, and entry-point authorization assumptions. No direct database access.
Tool Presents bounded operations to the agent layer. Must preserve consent and delegation context.
Operon Executes business logic, analysis, fetchers, and storage mediation. Impure operons require validated authority context.
Service Owns database and storage access. Routes and agents do not bypass the service layer.

This structure is already sufficient for partner-facing agent architecture discussions because it gives Hushh a clear execution model, clear policy boundaries, and a reproducible way to extend into additional agent families over time.

3.2 Sub-Agent Specifications

Kai currently expresses the most mature specialist-agent design in the repository. Its financial committee is implemented as a three-specialist debate framework backed by structured storage and streaming diagnostics.

Specialist Role Current Input Surface
Fundamental Agent Analyzes filings, fundamentals, and company quality. World-model context plus filing and financial inputs in hybrid paths.
Sentiment Agent Analyzes news flow and market narrative. News and context signals with explicit degraded-state behavior when providers are partial.
Valuation Agent Assesses metrics, relative pricing, and investability. Financial domain context plus market data inputs.
Debate Engine Synthesizes specialist output into explainable decisions. Produces typed decision payloads, transcript views, and observability diagnostics.

3.3 Agent Deployment on MuleSoft

The present Hushh runtime executes within FastAPI services and related Python processes, and that remains the operative deployment model. MuleSoft belongs in this document because it is the appropriate future enterprise orchestration layer for taking governed Hushh workflows into broader partner environments without redesigning the private-data model.

That alignment works precisely because the current system already separates agent logic, workflow state, and private data. In other words, the engineering precondition for MuleSoft alignment exists today, even though MuleSoft is not yet the active production host of the agent stack.

4. MCP Server Specifications

4.1 MCP Server Registry

Hushh already exposes a working MCP server over stdio JSON-RPC 2.0. It is not a conceptual placeholder. The current server exposes consent, discovery, domain, delegation, and IAM-related read surfaces that allow external tools to interact with Hushh without bypassing policy.

Surface Current Capability Family Purpose
Consent tools request_consent, validate_token, check_consent_status Drive approval and validity checks for external hosts.
Scope discovery tools discover_user_domains, list_scopes, /api/world-model/scopes/{user_id}, and /api/v1/user-scopes/{user_id} Return the user-specific scope catalog derived from available_domains, domain_summaries, and domain_registry.
Domain access tools Generic read surfaces bound to approved scope strings rather than a fixed domain catalog Return only the approved branch of user data for the exact dynamic scope that was granted.
Delegation tools delegate_to_agent and TrustLink creation surfaces Create TrustLinks for controlled agent-to-agent delegation.
MCP resources hushh://info/server, hushh://info/protocol, hushh://info/connector Provide self-description and machine-readable usage guidance.
RIA read-only tools list_ria_profiles, get_ria_profile, get_ria_client_access_summary Support the Investor and RIA runtime surface without granting write authority.

4.2 MCP Server Implementation Standard

MCP access in Hushh is deliberately dynamic. Hosts do not request against a hard-coded universal permission list and they do not assume a fixed set of domains. They discover actual scope strings for a specific user through discover_user_domains(user_id), /api/world-model/scopes/{user_id}, or /api/v1/user-scopes/{user_id}, where the backend derives scope availability from world_model_index_v2.available_domains, sanitized domain metadata, and domain_registry.

  1. Discover the user's current domain and path vocabulary from the world-model index and registry rather than from a baked-in client list.
  2. Request explicit consent for the exact scope family needed: attr.{domain}.*, attr.{domain}.{subintent}.*, attr.{domain}.{path}, world_model.read, world_model.write, or an appropriate agent.* authority when the operation is agent-executed.
  3. Approve through the Hushh product surface, where the user authorizes the export path.
  4. Consume only the encrypted scoped export created for that approval and time window.
  5. Respect revocation, expiry, and policy failure as first-class outcomes.

vault.owner remains the owner-authority control-plane scope used for protected vault and world-model operations. It is not a replacement for narrow third-party data scopes.

Production delivery is FCM-first, which keeps the approval step tied to the Hushh app rather than an ungoverned polling experience.

5. A2A Protocol Implementation

5.1 Hushh Agent Card Specification

Hushh's A2A surface is anchored by concrete agent metadata, not by vague delegation claims. Kai already exposes a manifest with identity, required scopes, optional hybrid scopes, capabilities, and compliance posture.

Agent Card Element Current Kai Definition Meaning
Identity agent_kai, version, description Stable machine-readable identity for discovery and routing.
Required scopes world_model.read, world_model.write, agent.kai.analyze Kai operates only with explicit private-data and agent authority.
Optional hybrid scopes external.sec.filings, external.market.data, external.news.api Live external data remains optional, named, and consent-aware.
Capabilities Hybrid, real-time, historical, and on-device-compatible capability metadata Signals the intended execution envelope of the agent architecture.
Compliance Educational-only, disclaimer-required, audit-trail-aware Bounds how Kai should be represented in product and partner contexts.

5.2 A2A Request/Response Flow

1. An originating agent or host requests a delegated action.
2. Hushh creates a TrustLink containing user, source agent, target agent, scope, and TTL.
3. The target agent verifies the TrustLink cryptographically.
4. Operons execute only if the inherited authority satisfies the requested scope.
5. The action completes without enlarging the original user's approved boundary.

That last point is the operative A2A rule in Hushh. Delegation carries authority forward; it does not manufacture new authority.

As the trust fabric matures, the same handshake can become value-carrying. A delegated access event can eventually attach bid or settlement metadata so that every approved retrieval of personal information from the world model can resolve into a background user credit, including minimum micro-credit patterns such as 0.01 cents to a future Hushh coin ledger, without turning the consent flow into a visible marketplace ceremony.

7. Security Architecture

7.1 Authentication Flow

Hushh does not treat authentication as equivalent to authorization. A signed-in user still needs explicit owner authority before the platform will perform protected data or agent operations.

Firebase Sign-In
  -> /api/consent/vault-owner-token
  -> VAULT_OWNER token
      -> world model operations
      -> Kai operations
      -> consent approval and revocation
      -> scoped delegation into MCP and A2A paths

This token hierarchy is one of the key architectural truths to preserve in any external systems conversation.

7.2 Encryption Standards

Hushh stores private user content as AES-256-GCM ciphertext only. The backend never persists plaintext private data. Vault wrappers represent alternate unlock methods for the same underlying DEK rather than alternate data protection models.

  • world_model_data stores ciphertext, IV, tag, and algorithm metadata.
  • world_model_index_v2 stores sanitized discovery and summary metadata only.
  • Passphrase and recovery wrappers are mandatory. Quick methods add wrappers; they do not weaken storage rules.
  • Zero-knowledge exports are created per consented slice and do not expose the full private blob.

This split is the engineering reason Hushh can discuss Salesforce-readable workflow alignment without conceding the private data system of record.

7.3 Zero-Trust Principles

  1. BYOK is mandatory; the backend stores ciphertext only.
  2. Signed in is not consent. Every protected operation requires valid scope and token context.
  3. Tri-flow parity prevents insecure platform-specific shortcuts.
  4. Public discovery, verification, and workflow data remain separate from encrypted private data.
  5. Dynamic scope generation favors least privilege over broad static grants and is resolved from runtime metadata, not a fixed domain enum.
  6. attr.{domain}.* grants only that domain branch, and attr.{domain}.{subintent}.* grants only that sub-branch.
  7. world_model.read is full read authority across domains, while world_model.write is write authority for governed world-model operations.
  8. vault.owner remains owner authority; agent.* scopes remain action authority; neither should be confused with generic third-party read permission.
  9. Delegation inherits scope; it does not expand scope.

The current scope vocabulary already reflects this discipline: attr.{domain}.*, attr.{domain}.{subintent}.*, attr.{domain}.{path}, world_model.read, world_model.write, vault.owner, and relevant agent.* scopes.

The same rule set also governs future economic handshakes. Any credit, bid, or Hushh coin settlement must remain downstream of consent, bound to the approved scope, and attributable to the exact access event that touched the user's private data.

9. CI/CD Pipeline

9.1 Pipeline Architecture

Hushh uses a path-filtered tri-flow CI pipeline with a single aggregate status gate. The intent is to keep the delivery system strict enough to protect the platform while avoiding a sprawl of semi-duplicated checks. Secret scanning, web checks, backend protocol checks, and integration checks are the default blocking lanes. Advisory lanes cover docs parity, subtree drift, native release readiness, and environment parity.

9.2 Deployment Environments

Environment Role Engineering Notes
Development Local backend and local or staged client profiles Backend starts first; native plugins call the Python backend directly for parity.
UAT / staging Primary branch-level verification surface Current observability model splits staging and production cleanly.
Production Cloud Run deployed services with DB governance and backup posture gates Developer API and consent SSE defaults tighten under regulated production settings.

9.3 Deployment Checklist (Every Release)

  1. Verify routes, parity, cache, docs, and pre-launch audit behavior.
  2. Verify secrets parity and environment configuration.
  3. Run backup freshness and migration governance checks for production deploys.
  4. Deploy backend, then frontend, then align CORS and service configuration.
  5. Run health checks and smoke the core investor, consent, and IAM flows.

11. Error Handling Standards

11.1 Error Categories

Error Class Handling Standard
Identity and bootstrap failure Protected operations remain blocked until owner authority is established.
Consent denial, expiry, or revocation Access ends immediately and does not silently fall back to unauthorized data.
IAM compatibility failure Return compatibility-safe investor payloads or explicit IAM_SCHEMA_NOT_READY behavior.
Provider degradation Serve cache-aware or degraded responses with explicit labels rather than pretending freshness.
Import or analysis validation failure Emit diagnostics and terminal failure states instead of soft success.
World-model drift Use reconciliation and audit tools, not ad hoc record mutation.

11.2 Retry Policy

Retry behavior is allowed only where idempotence and policy permit it. Cache-backed reads and provider fetches may retry under cooldown rules. Consent approvals, vault writes, actor-state transitions, and export creation are not treated as blind retry candidates. Stream paths may degrade or terminate with diagnostics rather than replaying hidden operations behind the user's back.

13. Engineer Onboarding (Day 0 Setup)

Day 0 onboarding assumes Node 20+, Python 3.13, Firebase credentials, and Supabase or local Postgres connectivity. Engineers are expected to run make setup, use a Python .venv, configure backend DB_* and auth secrets, configure frontend Firebase and backend URL values, and bring up the backend before starting client surfaces. The setup path is intentionally documented as a governed environment, not as a loose local sandbox.

  1. Clone the repository and run setup hooks.
  2. Install frontend and backend dependencies.
  3. Configure backend environment and client environment values.
  4. Initialize schema and start backend first.
  5. Start web or native clients.
  6. Verify health, login, vault creation, and owner-token issuance.

14. Operational Runbooks

14.1 Incident Response

Operational triage begins by locating the affected route, request ID, environment, and product plane: auth, consent, world model, provider, IAM, or marketplace. Because the platform is intentionally divided into private, workflow, and agent layers, incident response can usually isolate failures to a much tighter subsystem than would be possible in a flat application architecture.

14.2 On-Call Rotation

The current repository documents ownership expectations, release gates, observability, and recovery posture, even though it does not yet encode a large-team formal rota within the codebase itself. The present operating model is disciplined and evidence-driven, with a clear path toward more formal enterprise rotation practices as the platform surface expands.

14.3 Rollback Procedure

Rollback follows the same governance path as deployment. Cloud Run service history, route and parity integrity, secrets parity, migration release manifests, and backup posture checks determine whether a rollback is safe. Backend rollback is never treated as a code-only concern when schema drift or production data posture is involved.

14.4 Data Recovery

Production recovery posture is based on daily logical backups, freshness gates, release evidence manifests, and restore drills. The current documented recovery profile targets an effective RPO of no more than 24 hours, an RTO of no more than 2 hours, and 14 days of retained backup history.

Recovery Parameter Current Target
RPO <= 24 hours
RTO <= 2 hours
Retention 14 days
Governance Freshness checks, migration guard, release manifests, and monthly restore drills.
