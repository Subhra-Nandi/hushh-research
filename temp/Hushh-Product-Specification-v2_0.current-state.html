Current Hushh Edition | Salesforce CRM + MuleSoft Direction | March 2026 | Hushh Research | CONFIDENTIAL

This edition keeps the original product specification language and structure, while aligning it to the current Hushh product: Kai is live as the flagship surface, the world model is the private system of record, and the Investor plus RIA trust layer is the current path into broader platform expansion.

1. Executive Summary

Hushh is a personal agent that is meant to know enough about the user that nobody they trust has to keep asking the same questions again. In the current product, that promise is already visible in practical form: Kai assembles and uses private context from the encrypted world model, consent governs every protected data movement, and investor, advisor, marketplace, and profile workflows are beginning to sit on top of the same trust architecture.

Hushh does not yet run every enterprise interaction through Salesforce Agentforce and MuleSoft Anypoint Platform, nor does it yet automate every domain end-to-end. What it does have now is the part that matters most: a user-owned private data plane, an actor-aware workflow plane, and an agent surface that can already act with memory, policy, and explainability. Apple and Google remain the trusted consumer and agent platforms where Hushh lives. The encrypted world model is the present personal data core. Salesforce CRM and MuleSoft remain the intended enterprise workflow, governance, and orchestration layer as Hushh broadens its reach.

2. Product Vision

2.1 The Promise

You tell Hushh who you are once. Hushh handles everything else forever.

That promise begins with a simple engineering truth: a user's facts, preferences, and active intent should not need to be repeatedly reconstructed across every interaction. In the current Hushh product, that continuity already exists for Kai onboarding, portfolio intelligence, consented sharing, and the emerging advisor workflow layer.

2.2 What the User Controls

Profile - The facts. In the current product those facts are stored as encrypted world-model data, actor state, and safe workflow projections rather than as a flat CRM card.

Preferences - How they like things. Today these preferences are already domain-scoped, summarized safely for discovery, and available for reuse through consented flows.

Intent - What they need right now. The active intent is already visible across onboarding, portfolio import, analysis, marketplace discovery, and RIA workflows.

Consent - The user's terms. Who can see what, under what scope, for how long, and for what purpose remains the legal and emotional foundation of the product.

2.3 The Two Agents Every User Gets

Every Hushh user is being provisioned toward two core agent roles.

Agent Kai (the doer) - Kai is the active flagship product surface today. Kai collects and organizes private context, assembles the user's working profile, and helps them complete high-value tasks across onboarding, import, market intelligence, analysis, and optimization without asking for information that already exists in the vault and world model.

Agent Nav (the shield) - Nav remains the privacy and consent guardian of the platform. In the current product, Nav's responsibilities are carried through the vault, consent center, persona routing, marketplace trust surfaces, and relationship gating flows. Over time these duties will mature into a more explicit agent surface, but the product function already exists: the user should feel powerful, never paranoid.

That is why Salesforce CRM and MuleSoft can be treated as future enterprise operating layers without forcing Hushh to abandon its encrypted private-data model.

The current Hushh repo does not yet run on MuleSoft, but it is already shaped in the way MuleSoft expects: routes are governed, agent interactions are policy-aware, workflow objects are explicit, and external connectivity is modeled through MCP, A2A, and gateway-like boundaries rather than through uncontrolled direct calls.

3.3 Salesforce CRM as Personal Data System of Record

In the current Hushh product, the encrypted world model is already the private personal data system of record. That is the architectural truth. Salesforce CRM belongs in this specification because it is the correct enterprise-facing system for projected relationship, consent, and activity workflows around that private core.

In practical terms, the Hushh data model already contains the pieces that map cleanly into Salesforce language:

  • Contact and actor records for persona, identity, and marketplace state.
  • Preference and profile state inside the world model, with safe summaries in the index layer.
  • Consent records for who has access to what, for what purpose, and for how long.
  • Connection and relationship workflows for advisors, invites, and trusted counterparties.
  • Activity history through consent and interaction auditing.

The inversion remains the same as in the original founder framing: the user owns the record about themselves, and enterprises receive selective access on the user's terms.

That workflow layer is also where future handshake economics belong. The product is being shaped so that the moment a trusted counterparty receives a user-approved slice of the world model, the event can be logged not only as consent and activity, but also as value created on the user's terms.

4. Protocol Stack

The Hushh agent operates across a small set of interoperable protocol layers. These are not random integration choices; they define how the platform will scale from current Kai workflows into a broader agentic network.

Protocol Owner Function in Hushh Current Status
MCP Anthropic ecosystem standard Agent-to-tool connectivity and governed external access to user-approved data. Implemented in the repo today.
A2A Google / Linux Foundation ecosystem Agent-to-agent communication, discovery, and delegated work. Implemented in part through TrustLinks, Kai A2A server support, and compliance checks.
UCP Google Future universal commerce execution path for checkout, payments, and loyalty. Directional product pillar, not current core runtime.
App Intents Apple Native Siri invocation of Hushh capabilities. Target consumer interface layer, not current primary control surface.

Today, the protocols meet in a practical way: world-model scopes are generated dynamically, MCP can request consented access, A2A delegation preserves boundaries through TrustLinks, and route-governed services keep policy centralized. As enterprise governance matures, MuleSoft becomes the natural place to host and route more of that protocol traffic.

Over time, that handshake layer can also become value-carrying. When Kai or another approved agent asks for personal information from the user's own data system, the world model, the approval event can carry background bid or credit metadata so the user quietly accrues value every time their data works for them.

5. Agent Architecture

Agent Kai is built on Google ADK as the orchestration framework and is the current flagship Hushh product. In the present repo, Kai is the active runtime proof point for what the platform means when it says the agent should already know enough to act.

Kai.Travel Travel Directional domain pattern carried forward from the founder specification. Future airline, hotel, booking, and loyalty connectors.
Kai.Commerce Commerce Directional domain pattern for future UCP-driven shopping and purchase flows. Future UCP, merchant, loyalty, and receipt connectors.

That shape matters because new domains can be added without changing the core Kai orchestrator. The current finance-first release is the proving ground rather than the terminal scope of the product.

5.2 Agent Nav - Detailed Design

Agent Nav is the consent and privacy enforcement layer of the Hushh product. In the current implementation, Nav's duties are distributed across the consent center, vault authority, persona routing, relationship gating, and audit surfaces. The product function already exists even before a fully separate Agent Nav runtime surface becomes the dominant expression.

Nav's job remains the same as in the founder specification:

  • Consent evaluation before protected data is shared.
  • Data minimization so only the approved slice leaves the private plane.
  • Activity feed integrity so the user can see what happened and why.
  • Trust scoring and relationship judgment as the network of counterparties grows.
  • Infrastructure enforcement so policy is not dependent on a single app screen behaving correctly.

As Nav becomes a more explicit product surface, it is also the natural guardian for handshake economics. The user should not need to negotiate value manually. Nav should be able to ensure that if an approved agent interaction creates value from personal context, the corresponding credit is tracked quietly in the background against the user's account.

8. End-to-End Data Flow Example

User logs into Hushh, imports a brokerage statement, and asks Kai for a portfolio decision.

  1. Kai onboarding captures the user's starting profile and persona.
  2. Portfolio import parses the statement, quality-gates the result, and writes the encrypted financial domain into the world model.
  3. Kai assembles the user's financial context from the world model and safe index summaries.
  4. Agent Kai runs committee-style analysis across its specialist reasoning paths.
  5. The result appears as a dashboard and analysis surface with receipts, history, and degraded-state awareness if any provider is partial.
  6. The user then enters marketplace or RIA flows and receives a scoped access request from an advisor.
  7. Consent is evaluated and approved or denied under explicit scope and duration rules.
  8. Every action lands in the activity and consent history surfaces while private data remains in the encrypted personal data plane.

Total user input: a handful of focused steps. Total repeated forms: minimized. The product goal remains the same as in the founder specification: the user should express intent, not rebuild their identity from scratch every time.

As the network expands, this same flow can become economically expressive without becoming more complicated. Each approved handshake can eventually resolve into a background micro-credit when Kai or another trusted agent retrieves governed personal information from the world model.

10. Security & Privacy

Privacy is not a feature. It is the product. Hushh enforces that principle at the storage layer, the consent layer, the workflow layer, and the eventual enterprise orchestration layer.

  • Encryption - AES-256 class storage rules, TLS in transit, native secure storage, and no plaintext fallback.
  • Zero-Knowledge Cloud - The world model stores ciphertext, not raw private user facts.
  • Defense-in-Depth Consent - Consent is enforced through product logic, token logic, and infrastructural boundaries, not a single UI checkbox.
  • Compliance Direction - Regional privacy, deletion, audit, and data-residency obligations are already shaping the product model.
  • Right to Deletion - The platform is designed so that the user can erase private, workflow, and connection state coherently.
  • Audit Trail - Data sharing, approvals, revocations, and agent interactions belong in immutable history surfaces.

11. Go-to-Market

11.1 Consumer Acquisition

The current Hushh product enters the market through Kai and investor trust flows rather than through the original travel-first launch story. That is the operative GTM correction. The logic, however, remains the same: start where the pain is visceral, the fragmented data is valuable, and the product can remove real friction immediately. Today that proof point is private investing intelligence and advisor relationship control. Travel and hospitality remain strong future vertical expressions of the same system.

11.2 Enterprise Revenue Model

The consumer product remains the trust-acquisition surface. Enterprise value comes from the workflow and network side of the system:

  • Consent-verified profile and workflow access that is materially more valuable than brokered or inferred data.
  • Agent hosting and orchestration paths for trusted counterparties and future brand agents.
  • Revenue participation in future agent-executed commerce flows.
  • Connector and workflow infrastructure licensing as the enterprise layer matures.

A core long-term product principle is that the handshake itself should reward the user. Where a consented access event creates enterprise value, Hushh can track a minimum background credit, starting from micro-credit amounts such as 0.01 cents, into the user's future Hushh coin balance. The point is not speculative token language. The point is that whenever Kai, or another agent acting through the same trust layer, gets personal information from the user's own database, the world model, the system should remember that value flowed from the user and should account for it accordingly.
